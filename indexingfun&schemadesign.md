# Indexing Fundamentals & Schema Design for AI Features

---

# Indexing Fundamentals

Without indexes, a database must perform a **collection scan**—reading every single document to find a match.

Indexes create an ordered, easily searchable data structure (typically a **B-Tree**) that points directly to document locations.

## Single Field Index

Speeds up queries looking up a single key.

```javascript id="8h7w1k"
db.collection.createIndex({
  email: 1
})
```

---

## Compound Index

An index on multiple fields.

```javascript id="4mr9zp"
db.collection.createIndex({
  status: 1,
  date: -1
})
```

**Crucial Rule:** Order matters!

Follow the **ESR rule** when ordering fields in a compound index:

* **E** → Equality
* **S** → Sort
* **R** → Range

# Schema Design for AI Features (Vector Search & LLMs)

Modern applications frequently integrate AI features like:

* Semantic Search
* Recommendation Engines
* Retrieval-Augmented Generation (RAG)

This requires storing **Vector Embeddings**—arrays of floating-point numbers generated by machine learning models (like OpenAI, Cohere, or Hugging Face) that represent the semantic meaning of data.

---

# The AI-Ready Document Schema

When designing schemas for AI, you must store the original metadata alongside the high-dimensional vector.

```json id="2cwjma"
{
  "_id": "kb_doc_452",
  "title": "How to reset your routing password",
  "category": "IT Support",
  "tenant_id": "company_abc",
  "content": "To reset your router password, locate the physical reset button on the back...",

  "chunk_index": 0,

  "embedding": [
    0.0125,
    -0.0432,
    0.8912,
    "...",
    -0.1204
  ]
}
```

# Key Considerations for AI Schema Design

## Chunking Text Strategy

Large documents (like an entire PDF manual) cannot be embedded as a single vector effectively.

You must break them down into smaller **chunks** (e.g., 500 characters with an overlap).

Your schema should treat each chunk as an individual document, mapping back to a parent document ID.

---

## Vector Indexing (ANN)

Standard **B-Tree** indexes cannot index vector arrays for semantic similarity.

You must use specialized **Vector Indexes**, such as:

* Hierarchical Navigable Small World (HNSW)
* Inverted File (IVF)

These perform **Approximate Nearest Neighbor (ANN)** search using distance metrics like:

* Cosine Similarity
* Euclidean Distance

---

## Hybrid Search Indexing

AI search is best when combined with traditional filtering.

For instance, if a user searches for:

> "How do I fix my router?"

You only want to search documents belonging to their specific enterprise.

Example:

```text id="s7q2xn"
tenant_id: "company_abc"
```

### Design Pattern

Ensure your vector database index supports **Pre-filtering**.

This filters the dataset by metadata keys **before** performing the heavy mathematical vector similarity calculations, saving massive compute costs.
